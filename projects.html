ESP32 Minecraft Damage Alarm

The idea: take an in-game event — my character taking damage in Minecraft — and make it physically real. Every hit registers as a real alarm going off next to me, not just a red flash on screen.

How it works

A Python script connects to the Minecraft server over RCON and polls for damage/health-change events on my player. When it detects a hit, it sends a signal over Wi-Fi to an ESP32, which is wired to a buzzer and LED. The ESP32 runs a small listener that triggers the alarm the moment it receives the packet, so the delay between an in-game hit and the real-world alarm is close to instant.

Why it was interesting

This was my first project bridging software running somewhere else (a game server) with a microcontroller reacting in real time. It forced me to think about networking on the ESP32 side and reliability — what happens if a packet drops, or the connection resets mid-session — rather than just wiring up a sensor and reading a pin.

What's next

I'd like to extend it to react to different event types (fall damage vs. mob damage vs. explosions) with different alarm patterns, and eventually move the alert logic onto the ESP32 itself so it doesn't depend on the Python bridge staying alive.

Reaction Time Game

The idea: a standalone hardware reflex tester — no laptop required. Press a button as fast as you can after a light comes on, and see your time in milliseconds on a physical display.

How it works

Built on an Arduino Mega 2560. A microcontroller-driven timer starts on a randomized delay, lights an LED, and begins counting. A button press stops the clock, and the result is written out to a 4-digit 7-segment display in real time. LEDs give visual feedback for "too early" versus a valid reaction.

Why it was interesting

This project was really about timing precision and display driving — getting the 7-segment display to update cleanly without flicker, and making sure the randomized delay couldn't be gamed by guessing the pattern. It was my first real exercise in reading and reacting to hardware inputs on a tight timing budget.
